      • mixin() ← Using Deep Copy
        • Technique for adding properties and methods of some objects to another object, without using inheritance, without creating an inheritance chain.
        • Basic mixin functionality is provided by the Object.assign() method. But there is a problem with this solution since it creates a shallow copy: Objects are copied only by reference.
        • A deep copy mixin() solution : 
          
function mixin(target,...objects) {
	for (const object of objects) {
		if(typeof object === 'object') {
			for (const key of Object.keys(object)) {
				if (typeof object[key] === 'object') {
					target[key] = Array.isArray(object[key]) ? [] : {};
					mixin(target[key],object[key]);
				} else {
				Object.assign(target,object);
				}
			}
		}
	}
	return target;
}
        • Usage example:
            const a = {}, b = { foo: 'bar' }, c = { numbers: [1,2,3] };
            mixin(a,b,c);
            << { foo: 'bar' , numbers: [ 1, 2, 3 ] }
        • Another uses:
          • Add a large number of properties to an object all at one.
          • To create a copy() method than can be used to make an exact, deep copy of an object.
          • Then, the copy function can be used to create a factory function.
          • To add modular functionality. This means: Encapsulate properties and methods in an object, then add them to other objects without the overhead of an inheritance chain being created.
          • Example: mixin(superman,flight,superSpeed,xRayVision);
        • this keyword usage:
          • Binding: The value of this points to the object calling a method.
          • Chaining functions: superman.fly().move().xray(); Methods can be chained together and called in sequence if they return a reference to this.
          • that = this is used to avoid problems with nested functions.
          • .bind(this); can be used for all functions to set the value of this inside the nested function.
          • for-of and arrow functions can be used to avoid the use of nested functions. Arrow functions should be used when anonymous functions are required in callbacks.

Vocabulary, Main Concepts

  • OOP:
    • Classes define blueprints of objects. Objects are then created as an instance of that class.
    • All objects are collections of key-value paired properties.
    • OOP properties:
      • Encapsulation: The inner workings are kept hidden inside the object and only the essential funcionalities are exposed to the end user.
      • Polymorphism: The same process can be used for different objects. Methods can be shared, but also overriden with a more specific implementation.
      • Inheritance: Take an existing object and inherit its properties and methods to a new one.
    • static keyword. In class declarations, with "static methods" we get a class method. These methods are called by the class directly rather than by instances of the class.
      Dice is a class. So static description() { ... } decribes what that class does, and not just what their instances do. Static methods are not available for instances.
    • The object prototype is live, so new properties and methods can be added to existing instances.
    • POLYMORPHISM:
      "The concept of polymorphism means that different objects can have the same method, but implement it in different ways. The Object.prototype object has a toString() method that is shared by all objects. This means every object created in JavaScript will have a toString() method. Polymorphism means that objects are able to override this method with a more specific implementation. So although every object has a toString() method, the way it’s implemented can vary between different objects. For example, calling it on an array object will return each value in a comma-separated string."ebook, JS: From Novice to Ninja
    • Monkey-patching is the process of adding methods to built-in objects by augmenting their prototypes. This should be done with caution as it can cause unexpected behavior in the way built-in objects work.
    • Composition over inheritance is a design pattern where objects are composed from “building-block” objects, rather than inheriting all their properties and methods from a parent class.

  1. addeventlistener() ⇗
    • document.addEventListener(event, function, useCapture)
      • event: Required. A String that specifies the name of the event.
      • function: Required. Specifies the function to run when the event occurs.
      • useCapture: Optional. A Boolean value that specifies whether the event should be executed in the capturing or in the bubbling phase.
        • true - The event handler is executed in the capturing phase.
        • false - Default. The event handler is executed in the bubbling phase.
